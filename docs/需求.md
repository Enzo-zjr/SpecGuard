# 智能代码助手平台 - 详细需求规格

## 一、 产品概述
本系统是一个面向开发团队的Web平台，其核心能力是允许团队上传自身的代码规范与架构设计文档，并基于此对提交的Java源代码进行自动化、智能化的评审与建议，旨在提升代码与团队规范的一致性。

## 二、 用户角色与核心用例

### 1. 普通开发者
*   **UC-01 提交代码评审**：开发者可以选择一个项目，并通过上传文件或粘贴代码的方式，提交一段Java源代码进行评审。
*   **UC-02 查看评审报告**：开发者可以清晰查看评审结果，包括发现的问题、引用的相关团队规范条款以及具体的代码修改建议。
*   **UC-03 查看历史记录**：开发者可以查看自己提交过的历史评审记录及其详情。

### 2. 项目/团队负责人
*   **UC-04 管理团队知识库**：负责人可以为团队或特定项目上传、更新或删除代码规范、架构设计等文档（支持纯文本和Markdown格式）。
*   **UC-05 管理项目**：负责人可以创建项目，并将项目与特定的知识库（规范集）关联起来。
*   **UC-06 查看项目合规概况**：（远期）查看项目代码整体与团队规范的符合度趋势。

## 三、 功能性需求

### 模块一：用户与项目管理
1.  **FR-001 用户认证与授权**
    *   系统须提供用户注册与登录功能。
    *   用户登录后，系统需维持其会话状态。
2.  **FR-002 项目管理**
    *   已登录用户可创建项目，项目属性至少包括：项目名称、描述。
    *   用户可查看、筛选和搜索自己拥有或参与的项目列表。
    *   在项目详情页，用户可以将该项目关联到一个或多个“规范集”。

### 模块二：团队知识库管理（核心）
1.  **FR-003 规范集管理**
    *   用户可创建、命名和描述“规范集”，用于归类相关文档。
    *   用户可将规范集与特定项目绑定。
2.  **FR-004 文档上传与存储**
    *   在规范集内，用户可通过Web界面上传纯文本（.txt）或Markdown（.md）格式的文档。
    *   系统需持久化存储文档的原始内容及元数据（如：文档名、所属规范集、上传者、上传时间）。
3.  **FR-005 文档内容处理**
    *   系统需对上传的文档进行预处理，将其按段落或章节分割为语义独立的文本片段。
    *   系统需为每个文本片段生成向量化表示，并建立索引，以支持基于语义的相似度检索。
4.  **FR-006 知识检索**
    *   当触发代码评审时，系统能根据被评审代码的语义，自动从关联项目的规范集中检索出最相关的若干条（如3-5条）规范文本片段。

### 模块三：智能代码评审（核心）
1.  **FR-007 代码提交与解析**
    *   系统需提供Web界面，供用户提交待评审的Java源代码（完整文件或片段）。
    *   系统需对提交的Java代码进行基础解析，能提取类、方法、变量名等关键结构信息。
2.  **FR-008 智能评审执行**
    *   系统需将（解析后的代码结构信息 + 检索到的相关规范片段 + 原始代码）组合成结构化的提示词（Prompt）。
    *   系统调用大语言模型（LLM）服务，提交上述Prompt，并获取模型生成的评审意见。
    *   系统需要求LLM的回复按指定格式组织，至少清晰区分：**违反的团队规范及具体条款**、**发现的通用代码问题**、**具体的改进建议或代码示例**。
3.  **FR-009 评审报告生成与展示**
    *   系统需将LLM返回的结果进行解析，并在前端以清晰、友好的格式展示。
    *   报告必须明确高亮显示引用了哪条团队规范。
    *   报告中的改进建议应可直接复制。
    *   系统需保存完整的评审请求与结果，供历史查看。

### 模块四：系统管理
1.  **FR-010 基础配置**：系统管理员可配置LLM服务的连接参数（如API端点、密钥）。

## 四、 非功能性需求
1.  **性能需求**
    *   从用户提交代码到完整显示评审报告，端到端响应时间在常规网络条件下应不超过30秒。
    *   知识库文档上传与处理过程应在后台异步执行，不影响用户界面响应。
2.  **可靠性需求**
    *   核心的代码评审功能可用性应不低于99%。
    *   当LLM服务不可用时，系统应给出明确的错误提示，而非无响应。
3.  **安全性需求**
    *   用户密码需加密存储。
    *   用户只能访问和操作其拥有或参与的项目及关联的知识库内容。
    *   所有API接口需进行身份验证。
4.  **可维护性需求**
    *   系统的提示词（Prompt）模板应设计为可配置，无需修改代码即可进行调整优化。
    *   关键业务逻辑应具备日志记录能力，便于问题排查。
5.  **易用性需求**
    *   Web界面应简洁直观，新用户无需培训即可完成上传文档和提交代码评审的基本操作。
    *   评审报告应易于阅读，关键信息一目了然。
